# ADR-005 — Estilo de API: REST com OpenAPI

**Status:** Aceito
**Data:** 2026-06-01
**Autor:** Equipe PrimeStudy

---

## Contexto

O PrimeStudy expõe uma interface programática consumida por um frontend web
(plano de entrega HTTP — `src/interface/`). O Trabalho Final (Seção 7) exige
escolher e justificar formalmente um estilo de API e especificá-lo. As forças:

- **Consumidor único e simples:** um frontend web próprio, sem necessidade de
  consultas agregadas complexas ou de múltiplos clientes heterogêneos.
- **Modelo de recursos claro:** o domínio expõe recursos bem definidos
  (`estudo`, `materia`, `checklist`, `conteudo`) com operações CRUD naturais.
- **Maturidade da equipe:** REST tem a menor curva de aprendizado e o melhor
  ferramental conhecido pelo time.
- **Necessidade de documentação formal:** o enunciado pede especificação
  (OpenAPI/SDL/.proto).

---

## Decisão

Adotar **REST sobre JSON**, especificado formalmente em **OpenAPI 3.0**
(`docs/openapi.yaml`).

Convenções adotadas:

- **Recursos e verbos HTTP:** `POST /api/estudos` cria, `GET` lista/detalha,
  `PATCH` renomeia, `DELETE` remove, sub-recursos para `conteudo`, `checklist`
  e `materia`.
- **Versionamento por header:** o servidor responde com `X-API-Version: v1`. Foi
  preferido a versionar por URL para manter as URLs estáveis e coerentes com os
  exemplos já citados em `docs/quality.md` (`/api/estudos/...`). Uma `v2`
  incompatível seria negociada por header, sem quebrar clientes.
- **Erros uniformes:** corpo `{ "erro", "codigo" }` com status HTTP semântico
  (400 validação, 401 autenticação, 404 inexistente, 502 dependência externa).
- **Paginação:** `limite` (1–100) e `offset` em listagens.
- **Autenticação:** sessão server-side por cookie assinado, obtida em
  `POST /api/sessao` (ver ADR-003).

---

## Consequências

### Positivas

- **Baixo atrito:** REST + JSON é universalmente conhecido pela equipe e por
  qualquer avaliador; o frontend consome com `fetch` direto.
- **Documentação navegável:** o `openapi.yaml` pode ser aberto em Swagger UI /
  Redoc, e foi validado contra as rotas reais (11 paths = 11 rotas de API, sem
  divergência), garantindo coerência documento↔código. A raiz `/` serve uma UI
  estática de demonstração e não faz parte do contrato da API.
- **Cacheabilidade e semântica HTTP:** verbos e status codes carregam
  significado, simplificando o cliente.

### Negativas / Trade-offs

- **Over/under-fetching:** REST pode retornar mais (ou menos) campos do que uma
  tela precisa. Mitigado pelo parâmetro `incluir_texto` no detalhe e por
  respostas enxutas na listagem.
- **Múltiplas chamadas:** montar uma tela rica pode exigir mais de uma
  requisição — aceitável para o volume do PrimeStudy.
- **Versionamento por header é menos visível** que por URL; documentado
  explicitamente no `openapi.yaml` para compensar.

---

## Alternativas Consideradas

| Alternativa | Por que rejeitada |
|---|---|
| GraphQL (SDL) | Resolve over-fetching, mas adiciona servidor de schema, resolvers e complexidade de cache desproporcional a um único frontend com necessidades CRUD simples. |
| gRPC (.proto) | Ótimo para comunicação serviço-a-serviço de alta performance; inadequado para um frontend de navegador (requer proxy gRPC-Web) e exagerado para o caso de uso. |
| REST sem especificação formal | Não atenderia a exigência da Seção 7 nem garantiria coerência documento↔código. |

---

## Vinculação com Atributos de Qualidade

Ver `docs/quality.md`:
- **Manutenibilidade** — contrato explícito e versionável; a borda HTTP não
  vaza para o domínio.
- **Segurança** — autenticação por sessão e convenção de erro que não vaza
  detalhes internos (500 genérico).
- **Usabilidade (para quem integra)** — especificação OpenAPI navegável e
  previsível.

Relaciona-se com **ADR-003** (autenticação) e **ADR-002** (a camada `interface`
depende para dentro, mantendo o domínio livre de HTTP).
