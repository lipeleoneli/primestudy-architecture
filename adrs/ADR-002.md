# ADR-002 — Arquitetura Interna: Clean Architecture com influência Hexagonal

**Status:** Aceito  
**Data:** 2026-06-01  
**Autor:** Equipe PrimeStudy

---

## Contexto

O PrimeStudy (PI V) foi entregue como um monolito Flask com toda a lógica concentrada em `app.py`: rotas HTTP, regras de negócio e chamadas diretas ao Firestore e à API do Gemini no mesmo lugar. Isso criou dois problemas sérios:

1. **Testabilidade zero** — para testar a lógica de geração de conteúdo é preciso ter Firebase e Gemini configurados, pois a função chama `db.collection(...)` diretamente.
2. **Alto custo de mudança** — trocar o Firestore por outro banco, ou o Gemini por outro modelo, exigiria alterar as rotas Flask, que são a interface pública da aplicação.

O trabalho final exige demonstrar decisões arquiteturais conscientes e aplicar SOLID. A arquitetura interna precisa evidenciar esses princípios no código.

---

## Decisão

Adotar **Clean Architecture** (Robert C. Martin) como estilo interno, com influência de **Arquitetura Hexagonal** (Alistair Cockburn) na definição das interfaces (ports) que isolam o domínio da infraestrutura.

A estrutura em camadas é a seguinte, do núcleo para fora:

```
┌─────────────────────────────────────────────────────┐
│  interface/        Flask routes (HTTP, JSON)         │
├─────────────────────────────────────────────────────┤
│  infrastructure/   Firestore, Gemini, pdfplumber     │
├─────────────────────────────────────────────────────┤
│  application/      Use Cases, Strategies, Factory    │
├─────────────────────────────────────────────────────┤
│  domain/           Entidades + Ports (interfaces)    │
└─────────────────────────────────────────────────────┘
```

**Regra de dependência (Clean Architecture):** as setas de dependência apontam sempre para dentro. `domain/` não importa nada de `infrastructure/`. `application/` não importa Flask.

**Ports (influência Hexagonal):** `IEstudoRepository`, `IGeradorIA` e `IPDFParser` são interfaces Python (ABC) definidas em `domain/ports/`. As implementações concretas (`FirestoreEstudoRepo`, `GeminiAdapter`, `PDFPlumberParser`) vivem em `infrastructure/` e implementam essas interfaces. As rotas Flask recebem os use cases já montados via injeção de dependência.

---

## Consequências

### Positivas

- **Testabilidade:** os use cases podem ser testados com mocks de `IEstudoRepository` e `IGeradorIA` sem nenhuma dependência externa.
- **Substituibilidade:** trocar o Gemini por outro modelo de IA exige criar um novo adapter que implemente `IGeradorIA` — sem tocar nos use cases.
- **Aplicação direta de SOLID:** DIP é estrutural (domínio depende de abstrações), SRP é reforçado pela separação em camadas, OCP pelo padrão Strategy registrado na Factory.
- **Coerência com o enunciado:** a decisão arquitetural é justificada, documentada e demonstrável no código.

### Negativas / Trade-offs

- **Mais arquivos:** a mesma funcionalidade que antes era uma rota de 30 linhas agora envolve entity + port + use case + adapter + route. O custo é justificado pelo ganho em manutenibilidade.
- **Curva de aprendizado:** para um time iniciante, a separação entre port e adapter pode ser confusa no início.
- **Overhead para app pequena:** Clean Architecture tem mais valor em sistemas que crescem. Para o tamanho atual do PrimeStudy, parte da estrutura é antecipação intencional para fins didáticos.

---

## Alternativas Consideradas

| Alternativa | Por que rejeitada |
|---|---|
| MVC simples (Flask + Blueprint) | Não resolve o acoplamento com Firestore/Gemini; dificulta testes |
| Arquitetura em camadas clássica | Não impõe a regra de dependência; camadas superiores ainda poderiam importar infra |
| Manter o monolito original | Não demonstra os princípios exigidos pelo trabalho |
