# ADR-003 — Banco de Dados: Firebase Firestore + Firebase Authentication

**Status:** Aceito  
**Data:** 2026-06-01  
**Autor:** Equipe PrimeStudy

---

## Contexto

O PrimeStudy precisa armazenar:
- Dados de autenticação de usuários (email, senha, OAuth).
- Estudos e matérias, que são **estritamente por-usuário** — um usuário nunca acessa dados de outro.
- Conteúdo gerado pela IA, que é um campo de texto longo e não estruturado acrescentado ao documento de estudo progressivamente.

O time é de 5 pessoas, nenhuma com experiência em administração de banco de dados relacional em produção. O orçamento é zero (uso da camada gratuita).

---

## Decisão

Manter **Firebase Firestore** como banco de dados e **Firebase Authentication** para autenticação, conforme estabelecido no PI V.

A estrutura de dados no Firestore é:

```
usuarios/{uid}/
  materias/{materia_id}  →  { nome, cor }
  estudos/{estudo_id}    →  { nome, texto, conteudo{}, materia_id?, checklist[], criado_em }
```

A autenticação é feita no frontend via Firebase SDK (email/senha). O backend Flask verifica o `idToken` JWT na rota `/api/sessao` usando `firebase_admin.auth.verify_id_token()` e cria uma sessão Flask com o `uid`.

---

## Consequências

### Positivas

- **Isolamento natural por usuário:** a hierarquia `usuarios/{uid}/estudos` garante que as queries de um usuário nunca tocam dados de outro — sem necessidade de cláusula `WHERE user_id = ?` em todas as queries.
- **Schema flexível:** o campo `conteudo` é um mapa dinâmico; adicionar um novo tipo de conteúdo gerado (ex.: "podcast_script") não exige migration de schema.
- **Autenticação pronta:** Firebase Auth elimina implementação de hash de senha, recuperação de conta e tokens JWT — risco de segurança crítico resolvido por um serviço especializado.
- **Camada gratuita generosa:** Firestore oferece 50k leituras/dia e 20k escritas/dia grátis, suficiente para uso acadêmico.
- **SDK Admin para Python:** integração direta com Flask sem ORM adicional.

### Negativas / Trade-offs

- **Vendor lock-in:** migrar para PostgreSQL ou MongoDB no futuro exigiria reescrever `FirestoreEstudoRepo`. Esse risco é mitigado pela interface `IEstudoRepository` — o restante do código não muda.
- **Sem transações complexas:** Firestore suporta transações, mas não JOINs; a contagem de estudos por matéria exige query separada (`count()`), como já feito no PI V.
- **Custo em escala:** acima da camada gratuita, o custo por documento lido pode ser expressivo para aplicações com muitos usuários.

---

## Alternativas Consideradas

| Alternativa | Por que rejeitada |
|---|---|
| PostgreSQL (SQLAlchemy) | Exige servidor dedicado ou Heroku/Supabase; maior complexidade de deploy; schema rígido dificulta o campo `conteudo` dinâmico |
| MongoDB Atlas | Alternativa válida, mas o time já tinha Firebase configurado no PI V; migrar adicionaria risco sem benefício claro |
| SQLite | Não funciona em ambiente multi-processo/cloud; sem autenticação integrada |

## Vinculação com Atributos de Qualidade

Ver `docs/quality.md`:
- **Segurança / Confidencialidade:** dados isolados por UID; Firebase Auth gerencia tokens com expiração.
- **Manutenibilidade:** a interface `IEstudoRepository` permite trocar o Firestore sem impacto nos use cases.
- **Confiabilidade:** Firestore é gerenciado pelo Google com SLA de 99,99% — sem ponto único de falha sob responsabilidade do time.
